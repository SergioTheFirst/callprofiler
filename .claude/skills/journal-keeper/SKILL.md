# Skill: journal-keeper

## Назначение

AI-сессии над CallProfiler **не имеют сквозной памяти**. Между сессиями
остаётся только:
- код в репозитории,
- `CONSTITUTION.md` (merge-blocking правила),
- **`CHANGELOG.md`** — хронология изменений,
- **`CONTINUITY.md`** — состояние «где мы остановились».

Этот skill описывает жёсткий workflow работы с этими двумя журналами.
Он был введён по прямому требованию владельца:
> «CHANGELOG.md и CONTINUITY.md должны обязательно использоваться для
> записи твоих действий, чтобы ты ориентировался на прошлые шаги по
> принципу сервиса Obsidian. В конце выполнения каждого промпта ты
> должен заносить об этом метку в журнал, а перед выполнением нового
> знакомиться с этими журналами.»

## Когда применять

**ВСЕГДА**, в любой сессии над этим репозиторием, на трёх этапах:

1. **Старт сессии** — прочитать журналы.
2. **Перед `git commit`** — обновить журналы.
3. **При конфликте** — сверяться с журналами (раньше это уже делали?).

## Workflow

### Этап 1. Старт сессии (BRIEFING)

```
1. Read CONTINUITY.md (первые ~200 строк)
   → Узнать: ветка, прогресс, последний коммит, текущая задача,
     известные долги.
2. Read CHANGELOG.md (последние 100-150 строк)
   → Узнать: что именно исправлено/добавлено в недавних сессиях,
     чтобы не повторять работу и не ломать свежие фиксы.
3. При необходимости — CONSTITUTION.md (релевантная статья).
```

Только после этого — писать код или составлять план.

### Этап 2. Перед коммитом (LOGGING)

Для **каждой** значимой правки (не каждый тривиальный typo):

#### CHANGELOG.md

Добавить запись в `[Unreleased]` или под свежую дату `[YYYY-MM-DD]`.
Формат Keep a Changelog: `Added` / `Changed` / `Fixed` / `Removed` /
`Deprecated` / `Security`.

Шаблон:

```markdown
## [YYYY-MM-DD] — Короткий заголовок

### Fixed/Added/Changed — Краткое имя изменения (модуль.py)

**Проблема:** (если Fixed) что было не так.

**Цепочка:** (опционально) как воспроизводится.

**Исправление** в `module.py`:
\`\`\`python
# до
# после
\`\`\`

**Тесты:** добавлено N тестов, итого M pass.

**Результат:** что теперь работает / что гарантируется.
```

#### CONTINUITY.md

Обновить секцию «Текущее состояние» сверху:

```markdown
## Текущее состояние: YYYY-MM-DD (кратко что сделано)

### Ветка разработки
`claude/clone-callprofiler-repo-hL5dQ`

### Последний коммит
\`\`\`
<hash>  <сообщение>
\`\`\`

### Что сделано в этой сессии
- ...

### Следующий шаг
- ...

### Известные ограничения / долги
- ...
```

Плюс — добавить отдельную секцию «Сессия YYYY-MM-DD: ...» ниже,
с деталями для будущей себя/агента.

### Этап 3. Финальная проверка

Перед `git add`:

- [ ] `CHANGELOG.md` обновлён
- [ ] `CONTINUITY.md` обновлён
- [ ] Нет противоречий с `CONSTITUTION.md`
- [ ] Тесты прогнаны (`pytest tests/ -v`)
- [ ] Файлы с секретами (`.env`, токены) не включены

Коммит должен содержать **и код, и обновлённые журналы в одном комите**
(либо код в одном коммите + журнал сразу следующим, но до `git push`).

## Анти-паттерны

- **Коммит без обновления журналов.** Даже «мелкая» правка, если она
  осмысленная, должна оставить след.
- **Дублирование полной диффы кода в CHANGELOG.** Журнал — это «что и
  почему», а не `git diff`. Достаточно ключевых фрагментов.
- **Запись в CHANGELOG без даты или под чужой датой.** Сегодняшняя
  дата (в `env.today`) или `[Unreleased]`.
- **Стирание старых записей в CONTINUITY.md** при обновлении
  «Текущего состояния». Старые секции «Сессия YYYY-MM-DD» сохраняются
  вниз — это история.
- **Чтение только CHANGELOG без CONTINUITY** (или наоборот) в начале
  сессии. Оба файла нужны.
- **«Я пофиксю по-быстрому без журнала».** Нет. Быстрые фиксы — самые
  забываемые. Именно они — кандидаты на повторное внесение тех же
  багов.

## Быстрые команды

```bash
# В начале сессии:
# (Read tool)
# CONTINUITY.md offset=0 limit=200
# CHANGELOG.md offset=0 limit=150

# Перед коммитом — сверка:
git diff --stat
git log --oneline -5
pytest tests/ -v

# После обновления журналов:
git add src/ tests/ CHANGELOG.md CONTINUITY.md AGENTS.md
git commit -m "<короткий заголовок>

<тело с 'Почему' и 'Что'>

Journal updated: CHANGELOG.md + CONTINUITY.md
"
```

## Ссылки

- `CHANGELOG.md` — основной журнал (Keep a Changelog формат)
- `CONTINUITY.md` — журнал непрерывности (Obsidian-like)
- `CONSTITUTION.md` Статья 14 — правила разработки
- `AGENTS.md` раздел 3.1 и 3.3 — рабочий процесс AI-агента
