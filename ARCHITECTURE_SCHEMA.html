📊 CallProfiler

Архитектурная схема системы анализа телефонных звонков

🎯 Миссия: "Система много думает — показывает мало"

Тяжелый анализ оффлайн, пользователь видит только короткие диджесты.

📥 Слой 1: Входной поток (Ввод аудио)
Файловая система (D:\calls\audio)
Директория с аудиофайлами от Android-приложения. Форматы: WAV, MP3, M4A
Windows
Whisper (faster-whisper)
Транскрибирование аудио в текст. На GPU (~30 сек → 2-3 сек за звонок)
GPU 4.5GB
Pyannote 3.3.2 (Speaker Diarization)
Определение спикеров [me]=владелец, [s2]=другой. Загружает reference voice: ref/manager.wav
GPU 4.5GB
💾 Слой 2: Хранилище данных (SQLite)
callprofiler.db
Основная БД. 15 таблиц: calls, transcripts, analyses, contacts, promises, events, entities, relations, bio_*, narrative_*
SQLite3
FTS5 индекс (Full-Text Search)
Полнотекстовый поиск по транскриптам. Использует BM25 ранжирование
Поиск
Правило фильтрации: user_id
КРИТИЧНО: каждый SELECT/UPDATE/DELETE ДОЛЖЕН содержать WHERE user_id = ?
Безопасность
🧠 Слой 3: Анализ и обработка (LLM)
llama-server (Qwen 2.5 9B Q8)
Локальный LLM на http://127.0.0.1:8080 (OpenAI API совместимый)
GPU 10GB
Enricher (bulk_enrich)
Основной конвейер анализа: читает транскрипты → вызывает LLM → парсит JSON → сохраняет в DB
Pipeline
Промпт analyze_v001.txt
Система для извлечения: call_type, risk_score (0-100), promises, key_topics, entities
Версион
Memoization (bio_llm_calls)
Кэширование LLM ответов по MD5 хешу промпта. Избегает повторных вызовов одинаковых запросов
Оптимизация
🔍 Слой 4: Интеллект (Граф + Аналитика)
Knowledge Graph Builder
Агрегирует entities, relations, structured_facts из LLM ответов в граф знаний
Граф
Entity Metrics Aggregator
Вычисляет BS-index (Bullshit Index) по формуле v1_linear. Экспоненциальное затухание рисков за 90 дней
Риск
Psychology Profiler
Строит психологический профиль контакта: паттерны поведения, связи, рекомендации взаимодействия
Профиль
Contact Summary Cache
Быстрый доступ к сводке контакта: risk_score, promises, debt, facts. Обновляется после каждого анализа
Кэш
📖 Слой 5: Biography Pipeline (8-проходный)
Pass 1: Scene Extractor
Извлекает scenes из звонков (importance 0-100, tone, synopsis)
Pass 2: Entity Resolver
Канонизирует имена, мерджит aliases (Василий=Вася)
Pass 3: Thread Builder
Строит narrative threads для каждого контакта
Pass 4: Arc Detector
Обнаруживает многосцен арки (project, problem, relationship)
Pass 5: Portrait Writer
Пишет портреты контактов (3-5 абзацев, traits, relationship)
Pass 6: Chapter Writer
Пишет главы книги (2500-4500 слов, период=месяц)
Pass 7: Book Frame
Скелет книги: title, prologue, epilogue, TOC
Pass 8: Editorial
Редакция черновиков (убирает повторы, вычищает текст)
📤 Слой 6: Выходные каналы (Пользователю)
Telegram Bot
/digest - дневной диджест, /promises - открытые обещания, /contact - карточка контакта
Bot API
Android Overlay Cards
FolderSync → контактные карточки (риск, обещания, факты) на экране входящего звонка
Mobile
Web Dashboard (FastAPI)
Real-time SSE (Server-Sent Events) для live-обновлений. Polling fallback на 5-сек интервалах
Web UI
Book Export (Markdown)
Экспорт сгенерированной книги в markdown, PDF, EPUB (future)
Export
🔄 Основной поток данных (End-to-End)
1
Входящий звонок
Android записывает в D:\calls\audio\speaker_name_YYYYMMDD_HHMMSS.wav
2
Транскрибирование & Diarization
Whisper+Pyannote → JSON с сегментами [me]/[s2]/[?] текста
3
Сохранение в DB
calls.call_id, transcripts (с speaker, text, start_ms)
4
LLM анализ (bulk_enrich)
Промпт → LLM → JSON парсинг → analyses.raw_response (v2 schema)
5
Агрегация в граф
GraphBuilder → entities, relations, events, entity_metrics (BS-index)
6
Генерация контактной карточки
contact_summary → JSON (risk_score 🟢🟡🔴, promises, facts) → FolderSync на телефон
7
Telegram уведомление
Бот отправляет disjest (/digest), инлайн-кнопку feedback
8
Biography pipeline (по расписанию)
8 проходов → book_chapters, bio_books, markdown готовый к печати
📊 Основные таблицы базы данных
Таблица Назначение Ключевые поля
calls Метаданные звонков call_id, user_id, contact_id, call_datetime, direction, duration_sec, status
transcripts Текст звонка с говорящими call_id, speaker (OWNER|OTHER|UNKNOWN), text, start_ms
analyses LLM-анализ звонка call_id, call_type, priority, risk_score, raw_response (JSON v2)
contacts Контакты пользователя contact_id, user_id, display_name, guessed_name, phone_e164
contact_summaries Кэш контактной сводки contact_id, user_id, risk_score, promises (JSON), debt (JSON), facts (JSON)
promises Обещания/долги promise_id, contact_id, user_id, promise_text, deadline, status, fulfilled_at
events Событиях в графе (v2) event_id, user_id, entity_id, event_type, quote, polarity, intensity, schema_version
entities Сущности графа (люди, места, компании) entity_id, user_id, entity_type, canonical, aliases (JSON)
entity_metrics Вычисленные метрики сущности entity_id, user_id, bs_index, call_count, last_seen, formula_version
bio_scenes Biography pass 1 вывод call_id, user_id, importance, scene_type, synopsis, key_quote
bio_entities Biography сущности (отдельно от graph) entity_id, user_id, canonical, role, description, mention_count
bio_chapters Главы книги chapter_id, user_id, chapter_num, period_start, title, prose, status
bio_books Сгенерированные книги book_id, user_id, period_start, period_end, title, subtitle, book_type, version_label
⚠️ Критические ограничения (НИКОГДА не нарушать)
🚫 100% Локальная система
Нет облака, нет SaaS, нет интернет-зависимостей. Только локальное llama-server.
🔐 Фильтрация по user_id
КАЖДЫЙ SELECT/UPDATE/DELETE ДОЛЖЕН содержать WHERE user_id = ?. Это ЕДИНСТВЕННЫЙ способ изоляции данных пользователя.
🪜 Windows + Python (без Docker)
Среда: Windows 10, system Python, никаких контейнеров, никаких Redis/Celery. Запуск через batch files.
💾 SQLite3 + прямой SQL (без ORM)
Используй sqlite3 напрямую, никаких SQLAlchemy/Django ORM. Миграции через ALTER TABLE, не recreate.
🎯 GPU последовательность
Whisper+Pyannote (4.5GB) → выгрузка → LLM (10GB). Параллелизм невозможен, разные процессы.
🔑 Нет hardcoded токенов
os.environ.get() ТОЛЬКО. Telegram token, user_id — через ENV переменные.
📊 Выход ≤ 300 символов
User-facing output никогда не длиннее 300 символов или 3 фактов на item. System denotes a lot, shows little.
🛑 Никогда не глушить ошибки
Логируй + сохраняй в DB + продолжай. Исключения не должны ломать весь батч.
